## 1. State of the system

Core: S0→S7 staged pipeline, orchestrator-only coordination (sub-agents cannot spawn
sub-agents), human gates at S1 (angle) + S6 (sign-off), machine-checkable correctness as
the spine. Two content tracks routed at S0 (`factual_ai_news` full chain;
`aesthetic_lifestyle` shrinks the oracle to `aesthetic_audit.py`; `mixed_explainer`
RESERVED). `tools/run_oracle.py` is the mode→oracle dispatcher.

The verification stack (unchanged this iteration, all wired):
- **`tools/citation_audit.py`** — marker existence + freshness + word/keyword/must-cite +
  source-authority + banned-phrases. Exit 0/1. THE hard gate (S4, per section).
- **`tools/grounding_gate.py`** — faithfulness 1→2 and 2→3, cross-lingual, fails closed.
- **`tools/factcheck_gate.py`** — cited ≠ true antidote; any non-SUPPORTED claim → exit 1,
  fails closed on empty.
- **LLM-judge advisory** — S6 editorial review, screening only (now a 2-3 panel, see §2).

NEW coordination layer on top of (not inside) those gates:
- **`run_journal.jsonl`** per article + **`tools/journal.py`** (append/summary) — the
  append-only ledger of what RAN: dispatches, results, gate exits, human decisions,
  tokens/cost when known. Resume = reconcile result JSONs (what is GREEN) → journal
  (what RAN) → STATE.md (summary), in that order; STATE blind spots are backfilled from
  the journal. `tools/status.py` joins journal cost onto its matrix as a `cost` column.
- **S3 parallel waves** — all sections' write→fact-check→grounding→audit loops run
  concurrently; converge on every `sec<k>_audit.json` with `status:"pass"`.
- **S5.9 findings-triage (optional) + S6 panel** — one funnel between the gates and the
  human: triage dedupes + source-verifies findings (confident false positives →
  `reject`), then 2-3 independent reviewers vote; the orchestrator merges by majority.

Solid: the four-layer verification, per-stage result files + status matrix, the journal
TOOLING (fully static-verified incl. failure paths), the scaffolder, the xhs adapter
paths, CI (every gate has positive + negative fixtures; journal + cost column now too).

Partial / by-design-light: the parallel waves, the triage agent, and the panel merge are
DOC-level control flow for the orchestrator LLM — statically consistent, never yet
exercised by a live `/write-article` run. Token/cost capture depends on what the harness
surfaces after an Agent call ([ASSUMED], §6). S5.5 Gemini live API path still untested;
explainer oracle still unshipped; image render still Chrome-bound (prior threads, §4).

## 2. What changed this iteration (and why)

**A. Run journal (`tools/journal.py` + `run_journal.jsonl` + status.py cost column).**
WHY: STATE.md is a summary the orchestrator updates BETWEEN actions — a session dying
between an agent's return and the STATE write lost run history; dispatch counts (the
step budget) had no durable home across resumes; and spend was invisible until too late
(agent-fleet's 1.04M-token incident). Design decisions worth inheriting:
- Journals ORCHESTRATION events, not call-input hashes. Fleet's input-hash resume exists
  to decide "re-run or reuse this call"; here the per-stage result JSONs + oracle exits
  already answer that. What was missing was the ledger: who ran, what the gates said,
  what it cost, what the human decided.
- `append` fails fast (exit 2, NOTHING written) on per-event required fields and on a
  missing article dir — the fleet `validate_config` lesson (a typo must not become a
  silent no-op). `summary` tolerates malformed lines (a corrupt line must not brick
  resume) and prints per-section writer-dispatch counts = the step budget.
- NO price table in the tool. The orchestrator usually sees only a total token count for
  a sub-agent, not the input/output split, so any embedded pricing would be fake
  precision that also goes stale. Record `--cost-usd` only when a real number exists;
  the status cell falls back to `~Nk tok`, else `—`.
- `tools/status.py` output is byte-identical to before when no journal exists (CI-pinned)
  — old articles and fixtures are untouched. It imports its sibling `journal` module
  (script-dir import; fine for `python3 tools/status.py`, matters only if someone
  vendors the file alone).
- Orchestrator hard rule 7: journal AS IT HAPPENS, never batch-reconstruct, never edit
  past lines. Rules 1-6 unchanged.

**B. S3 parallel fan-out (orchestrator control flow + docs ONLY).**
WHY: sections are the smallest independently-verifiable units with disjoint files
(`sec<k>_*`) and independent contracts — serial execution was an accident of doc
phrasing, not a design need. Decisions:
- **Wave model**: each round, ONE parallel message dispatches every unconverged
  section's NEXT step (mixed steps across sections is normal). Within a section the
  chain stays strictly sequential; never two agents on one section per wave. This keeps
  the playbook deterministic for an LLM orchestrator — no speculative or overlapping
  dispatch to reason about.
- **Convergence = existence AND status**: every `sec<k>_audit.json` with
  `status:"pass"`. Existence alone lies — the auditor writes that file on FAIL too.
- Step budget (3 per section) unchanged; the journal makes it durable across resumes. A
  section hitting budget escalates ALONE; other sections keep going.
- Agent playbooks and oracles untouched, by mandate. `/write-section <k>` stays the
  sequential single-section path.
- Deliberately NOT done: running grounding 2→3 and the citation audit of the same
  section in one wave (they are independent reads of the same draft). Rejected to keep
  "audit is the last step of the chain" unambiguous; revisit only if wall-clock hurts.

**C. S5.9 findings triage + S6 review panel.**
WHY: agent-fleet's postmortem — 2 of 57 review findings were CONFIDENT false positives;
a single judge putting a wrong finding on the human's desk is the exact failure S6 had.
Decisions:
- **New agent `.claude/agents/findings-triage.md`** (S5.9, optional, factual track):
  aggregates every surviving finding/WARN from
  `sections/sec*_{writer,factcheck,grounding,audit}.json` + `stage_results/S5-humanize.json`,
  dedupes, **verifies each item against `must_cite` + the source pack BEFORE it reaches
  the table** (对照 must_cite/来源验真后再上桌), rejects confident false positives with a
  refutation (宁可驳回也不放行错误修改), ranks the rest `adopt`/`needs_editor` into
  `stage_results/S5-9-findings-triage.json`. It never edits, never re-opens
  oracle-green decisions, and adds NO new findings (it is a funnel, not a reviewer).
  Skipped (with a journaled `stage` skip event) when zero findings survived.
- **S6 = 2-3 independent `editorial-reviewer` variants in one parallel message** (2
  default; 3 when S2 said `complex` or triage left `needs_editor` items). Lens emphasis
  (argument/angle · narrative/AI 味 · audience/tone) weights DEPTH, does not narrow
  scope — every variant covers all axes so the majority vote stays meaningful. Each
  writes its own `S6-editorial-review-<variant>.json` (the spec now forbids the shared
  filename in panel mode — parallel writes to one file is last-writer-wins, the exact
  #15 bug class). Variants read the triage file and may not re-report a `reject` unless
  they can refute the rejection.
- **The orchestrator merges by majority** into the canonical `S6-editorial-review.json`
  (with a `panel` field) — the ONE exception to "agents write their own results",
  explicitly documented. Severity = majority; a lone BLOCKER downgrades to WARN, logged
  in DECISIONS.md, manually escalatable. Downstream (S7 precondition, status matrix,
  resume) is unchanged because the canonical filename is unchanged.
- Naming: prose says "S5.9"; every machine id (filename stem, JSON `stage` field,
  journal stage) is **`S5-9-findings-triage`** — status.py joins journal cost by file
  STEM, so a dotted machine id would silently break the join. Do not "fix" one side.

**D. CI repair — main was red since #23 (found while opening this PR).**
The `run_oracle` workflow step (added in #23) captured an EXPECTED exit 3 with a bare
`cmd; rc=$?`. The workflow shell is `bash -e`: any non-zero simple command outside a
condition context kills the step ON THE SPOT, so the run died at cmd (exit 3) and the
assert never executed. Every push to main since #23 failed there, and PR #24
(skeleton-unification; touches no test.yml) inherited the red. Fix:
`rc=0; cmd || rc=$?` — the `||` makes it a condition context, which `-e` exempts; the
two sibling `; var=$?` captures in the same step were hardened the same way, and an NB
comment in the step now names the trap so it is not reintroduced. Side effect worth
knowing: every step AFTER run_oracle (gemini_polish onward) had never actually executed
in CI — main died before reaching them. The full local replica (§5) confirmed they all
pass. **PR #24 needs no code change: update its branch from main after this PR merges
(`gh pr update-branch 24`) and its CI reruns green.**

## 6. Known risks / [ASSUMED]

- **Journal discipline is prose-enforced.** Hard rule 7 tells the orchestrator to append
  as it happens; nothing MACHINE-enforces it the way the oracles enforce gates. A lazy
  orchestrator produces a thin journal — resume then degrades gracefully to result JSONs
  + STATE.md (the pre-journal world), it does not corrupt.
- **[ASSUMED] the harness surfaces per-agent token usage** the orchestrator can copy
  into `--tokens-total`. Unverified until a live run; the schema treats usage as
  optional everywhere, so absence costs only the cost column's usefulness.
- **Parallel waves are only as safe as file disjointness.** Today every section-stage
  artifact is `sec<k>_*` and only the orchestrator writes STATE/DECISIONS/journal. Any
  future shared per-section artifact (e.g. a common glossary file writers append to)
  breaks the no-conflict argument — re-check the parallel-wave note in orchestrator.md
  before adding one.
- **Majority can downgrade a lone TRUE blocker.** Mitigations: the downgrade is logged
  in DECISIONS.md, the orchestrator may escalate manually, and the S6 human gate still
  reads the draft. Accepted trade: the fleet data said confident false positives are the
  more common failure.
- **Canonical S6 file is orchestrator-written** — the one exception to "agents write
  their own results". If a future refactor gives agents that write, the last-writer-wins
  bug (#15) comes back through the front door.
- **Naming seam:** prose "S5.9" vs machine id `S5-9-findings-triage` (filename stem =
  journal stage = JSON stage). Unifying them the wrong way breaks the status cost join.
- **Static-verified only** (repo tradition, worth repeating): waves, triage, panel merge,
  and live token capture have NOT run for real. The ITERATION_LOG entry marks the same.
- Prior risks from `-5` still stand: Chrome render nondeterminism, `MAX_BODY_CHARS`
  coupling, opt-in local commit guard, Apify cost/ToS, gitignored article content.
