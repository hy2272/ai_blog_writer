## 1. State of the system (unified)

A Claude Code multi-agent system that writes a verified Chinese post (AI-news OR 生活美学),
proving each unit against a machine-checkable oracle before advancing. Two structural ideas
now coexist:

**From #24 — "one skeleton, N fillings":** the pipeline is fixed; a `mode`/`track` picks the
fillings. S1 (`scout`) and S3 (`writer`) are **mode-agnostic shells** carrying no fixed
workflow; the orchestrator dispatches them with a `mode` + the exact path of a **mode-skill**
to Read and follow. Skills live in `.claude/skills/` (`tech-news-scout`, `aesthetic-scout`,
`tech-news-writing`, `aesthetic-writing`), each two-part (workflow + output-contract schema),
all `disable-model-invocation: true` (zero context load; Read-by-path is deterministic, not
description-matched). `research.md` + `aesthetic-writer.md` were removed (superseded by
`scout.md` + the writing skills).

**From #25 — the coordination layer above the gates:** an append-only `run_journal.jsonl` per
article (dispatches, gate exits, human decisions, tokens/cost); S3 runs ALL sections' loops as
parallel waves; S5.9 findings-triage + a 2-3 reviewer S6 panel funnel findings to the human.

Pipeline (both tracks share the skeleton; gates differ by track):
`S0 decompose + track router → S1 scout (mode-skill) → ⏸ angle gate → S2 editorial →
(grounding 1→2) → S3 write (parallel waves; per section: writer[skill] → fact-check →
grounding 2→3, factual only) → S4 citation audit (HARD, per section) → S5 humanize →
S5.5 Gemini polish → S5.9 findings-triage (optional) → S6 editorial-review PANEL ×2-3 →
⏸ human sign-off → S7 output`.

**Verification stack (the spine, unchanged by either feature):** `citation_audit.py` (marker
existence + freshness + word/keyword/must-cite + source-authority + banned-phrases, HARD S4) ·
`grounding_gate.py` (faithfulness 1→2 and 2→3, fails closed) · `factcheck_gate.py` (cited≠true,
any non-SUPPORTED → exit 1, fails closed) · `aesthetic_audit.py` (aesthetic track's shrunk
oracle) · `run_oracle.py` (mode→oracle dispatcher). LLM judgment sits ABOVE this, never
re-deciding what a gate decided.

**Solid:** the four-layer verification; both structural features present and self-consistent on
main; the journal tooling (fully static-verified incl. failure paths); the mode-skill split;
per-stage result files + status matrix + cost column; CI (39 steps, every gate has a
positive + negative fixture; journal + cost column now covered).

**Partial / not-yet-live:** NEITHER feature has been through a real `/write-article` run. The

DOC-level control flow for the orchestrator LLM plus statically-verified skills/tools — correct
on paper, unexercised in a real run. That single run is the top open thread (§4).
